Ideas for Decent Apps

 

Introduction

You’ll find many ideas for decent apps here. Each idea fits these criteria:

  • Could be implemented in 200 hours (13 weekends).
  • Fits the Decent architecture – local LLM, no service calls, running in a web browser.
  • Fair chance of being useful or entertaining to somebody.
  • Fits the guidelines for including an app on the Decent Apps portal.

You might look at an idea and think, “that idea isn't new at all. I've seen it in Product X”.

Of course, it’s very hard to do a brand new thing. Ideas build off previous ideas, and maybe your iteration of an existing idea brings new value. Or you end up making a privacy-respecting, free, open-source version of what already exists, and that is enough to earn you some user attention. And not everything needs to be unique to be valuable – there’s the value of showcasing your talent in a public space as well.

We encourage you to tweak these ideas how you like or even just use them as inspiration for new ideas of your own. There is no ownership of ideas here. Use as much or as little from them as you like.

Format of the Ideas

Here are the common elements that appear with each idea.

Element

Explanation

Name

Tends to describe what the app does rather than be a good name. You probably want to come up with your own app name.

Difficulty Level

How hard/risky it will be to write the basic functionality of the app within 200 hours.
  • Easy - we think a one-year coder could do it.
  • Medium - achievable by a software engineer with a few years of professional experience,
  • Advanced - achievable by a software engineer with 5+ years of pro experience and solid prompt engineeering skills.

Basic Functionality

A minimal set of features to release the app with. (aka “MVP)

Extra Feature Ideas

Some ideas going beyond the basic functionality to show how the app could be improved after its initial release.

Hints

Further thoughts to help in implementation

Category - Education

Name: Learn a Document

Difficulty: Medium

Basic Functionality:

This app quizzes the user on the contents of a document to help the user learn. Paste or upload some text into the app. Then the app generates questions about the doc and asks the user for answers. Each time the user provides an answer, they receive immediate feedback on the correctness of their answer.

Extra Feature Ideas:

  • A mastery score that increases as the user demonstrates more knowledge. The user aims to reach a certain mastery.
  • Coloring or marking the source document to show which areas the user understands well.
  • App notices weak areas and asks more questions around them.
  • Multiple choice or fill-in-the-blank questions that make it easier for the user to progress.
  • Load content from a web page URL. (Requires some intelligent scraping to get the intended context)

Name: Language Roleplay

Difficulty: Medium

Basic Functionality:

This app assists language learning for intermediate learners by having chats in the target language around role-played situations. For example, the user interviews for a job as a cook at a restaurant, and must answer some interview questions from the LLM who is roleplaying a hiring manager at the restaurant. Each situation has a desired outcome, e.g. “get hired as a cook”, and the LLM will have some criteria like “user must say the names of three kinds of foods” or “user must say which days of the week they are able to work”. The LLM will gently prod the user to meet these success criteria.

Extra Feature Ideas:

  • Some artwork showing the situation would be nice. E.g., show an illustration of a restaurant kitchen and the hiring manager along with the chat window.
  • Multiple lessons/situations organized by difficulty. User’s progress is tracked.
  • User can look up words they don’t know.
  • A way to get advice on improving the user’s responses, maybe after the user has completed the situation.

Name: Language Simplifier

Difficulty: Easy

Basic Functionality:

This app creates simplified versions of text that are more likely to include vocabulary and grammar familiar to a language learner. The user pastes/imports some text into the app and then requests simplification. The LLM generates a simplified version of the text in the target language for the learner.

Extra Feature Ideas:

  • User can specify a target level such as CEFR (A1/A2/B1/B2).
  • The app tracks or imports known versus unknown vocabulary and can make replacements based on it
  • Grammar-level simplifications can be made, e.g. "use only present tense" verbs
  • User can get definitions for words they don't know

Category - Work/Productivity

Name: Anonymizer

Difficulty: Easy

Basic Functionality:

This app takes freeform text provided by a user and rewrites it to remove any information or style that might identify the author of text. Useful for cleaning up anonymous survey responses.

Extra Feature Ideas:

  • Batch mode – import an XLS/CSV and anonymize each row
  • Parameters – UI shows what things user would like to anonymize (names of people, dates, etc.) and lets user tweak them.
  • Consequence analysis – some messages might have more consequences if the anonymity were lost for the author. For example, an angry rant about coworkers. These could be flagged for manual review.

Name: Source Code Evolver

Difficulty: Medium

Basic Functionality:

This app begins with a some small amount of code, maybe a few hundred lines, that is pasted/imported by the user. The user specifies some transformation to be made on the code, for example “add comments explaining what each function does” or “remove any functions that aren’t called by main()”.

Extra Feature Ideas:

  • Versioning – ability to revert back to any previous version and see differences between versions.
  • Unit tests – ability to check that the code hasn’t broken as various changes are made by running unit tests. (This is tricky to do safely, and needs some thought. LLM-generated code passed to `eval()` is not safe on its own.)

Name: Customer Feedback Classifier

Difficulty: Medium

Basic Functionality:

This app takes customer feedback for a product or service and classifies it as belonging to specific product features. So for example, if some customer feedback text is “The paper keeps jamming whenever I try to print.” the feedback could be classified as “paper feed tray” which is a category that can be applied to multiple items for organization, analytics, and routing. The user of the app can set up their category definitions and specify guidance for classifying, e.g. “problems with paper jamming are classified as ‘paper feed tray’ unless the customer mentions using glossy finish paper.”

Extra Feature Ideas:

  • Batch mode - import an XLS/CSV and run classification on each row.
  • Auto-categorize – generate a set of categories automatically by analyzing a collection of customer feedback for common themes across many feedbacks.

Name: Knowledge Card Expert

Difficulty: Advanced

Basic Functionality:

This app takes a question from a user and finds a knowledge card that correlates well to the question. A chat-based interface blends LLM-generated responses with knowledge cards that show pieces of factually verified information.

In this way, the user is protected somewhat from LLM hallucinations that might otherwise cause the user to be given incorrect advice. So for example, a user might ask “How much does the professional tier subscription cost?” and the LLM could display a card inside of the chat window with official pricing information, and maybe output some text after it like “I found this pricing information for the professional tier.”

Extra Feature Ideas:

  • Import knowledge cards from an XLS/CSV.
  • Scores relevance of multiple knowledge cards and presents multiple cards to user.
  • Gives answers that synthesize what was found on knowledge cards and presents its reasoning in the the chat session.
  • Can generate new knowledge card drafts that are unverified. A human must review these before they become verified.

Name: Link Manager

Difficulty: Advanced

Basic Functionality:

This app catalogs links to all the web pages you provide it and provides a better way of finding them again than the “Bookmarks Bar”. This is especially good for keeping track of work-related information. The user pastes a new link into the app. An LLM then works in the background to describe and index the page and its contents. Later, the user may find the link again with keyword searches that match the index.

Extra Feature Ideas:

  • There are many refinements that can be made to indexing and querying. For example, some keywords like “the”, “and”, “we”, etc. are common and should influence returned results less or not at all compared to more unique words.
  • Support for contiguous word (keyphrase) searches like “best practices” rather than finding documents that have “best” and “practices” in them.

Hints:

There are easy and hard ways to make the basic functionality work. Just to show how it can be done, let’s describe one of the easier approaches (still pretty advanced for a quick project):

  1. User pastes a URL.
  2. App fetches the content at the URL and parses it with a third-party library like Cheerio or JSDom into plaintext.
  3. Prompt the LLM to identify the areas of plaintext that look informational instead of UI and chrome (e.g. button text, navbar, footer links).
  4. Prompt the LLM to create a paragraph-long summary of the informational content and a title for the page. Store this metadata along with the URL as a link entry.
  5. Parse the doc for all unique words found and store them in a lookup index, where each keyword maps to one or more link entries.

Category - Personal Tools

Name: Family Tree Modeler

Difficulty: Medium

Basic Functionality:

This app receives information from a user in chat like “I have three brothers, Mike, Tom, and Larry.” and immediately updates a family tree graph to show each new piece of information.

Extra Feature Ideas:

  • Export graphs to an image format, e.g. SVG, PNG.
  • The app notices logical problems and mistakes and asks questions to clear them up, e.g. “Your brother Mike is also your father?”
  • App asks the user questions to fill out the tree, like “Do Shane and Emma have any kids?”

Name: Bibliography Generator

Difficulty: Medium

Basic Functionality:

This app generates bibliographies with citations with all correct formatting. The user only needs to paste freeform content about the publication. Useful for students or writers.

Extra Feature Ideas:

  • User provides URLs to Amazon web pages or other web pages that have info about the publication. (requires intelligent scraping)
  • Put rich-text formatted bibliography in clipboard so it can be pasted into a text editor like Word, Google Docs, etc.
  • Parameters for specifying a style (Chicago, APA, MLA, Harvard, AMA, etc.)

Name: EULA Checker

Difficulty: Medium

Basic Functionality:

This app reviews end user license agreements and calls out terms that seem non-standard or particularly important to the user. It can be used to check for potential privacy violations (selling data to third parties), IP ownership of data/ideas contributed by user, or other things. Some care would need to be taken not to give legal advice to the user, but rather present summaries with linkage to the original text.

Extra Feature Ideas:

  • User can specify what kinds of things are important to them with a simplified UI (e.g. checkbox for “privacy concerns”).
  • User can specify additional things to check for (e.g., “Anything that requires me to use mediation” or “Any penalties where I might owe them money for something”).
  • Ability to track previous versions of EULAs and show what has changed in updates.

Name: Places I’ve Been

Difficulty: Advanced

Basic Functionality:

This app tracks all the cities in the World a user has visited and can answer chat prompts about them. User clicks a “Where Am I Now?” button that uses the Geolocation API of the browser to perform a nearest-city lookup and then that city is added to their history. The user can later ask questions like “What cities have I been to in Germany?” and receive LLM-generated answers based on their travel history.

The nearest-city lookup can be done without service calls—just divide all the city data up into static-content chunks that cover geography and use HTTP GET fetches to retrieve a dataset of cities close to lat/long provided by Geolocation.

Extra Feature Ideas:

  • Display locations on a map. (Some work to figure out how to do this without sharing personal data, but it seems possible.)
  • Generate reports that show all the countries and cities a user has been to. Make them easily sharable on social networks for users to brag.
  • Track multiple visits to the same place over time.
  • User can add their own comments about each place. Or maybe retain a chat history that is specific to each city, so that past conversations with LLM about a city are retained.
  • LLM can generate flavor information about a city automatically. E.g., for Chicago the LLM might generate “Birthplace of the skyscraper and deep dish pizza.”

Name: Opinion Explorer

Difficulty: Advanced

Basic Functionality:

This app generates visual graphs that classify different opinions people can take on a given topic and why they might have them. The app becomes a quick way to see the wide range of possible opinions on a topic instead of just being exposed to a small number of views.

Imagine a question with some controversy to it like “Should we use more nuclear power for our energy needs?” The LLM generates a graph of potential answers. It might first be broken down into high-level “Yes”, “No”, and “Maybe” categories. And each category can break down with refinement into more nuanced positions, like under “No”, we might have “We may create a dangerous anomaly not currently understood by scientists”, “We don’t have a sustainable way to store nuclear waste”, or “The sites make attractive targets for military and terrorist attacks”.

Extra Feature Ideas:

  • Chat-based conversations between user and LLM that reference the generated opinions as context.
  • User can directly edit the graph and add their own nodes to it.
  • User can designate some opinions as carrying more weight to them and dismiss others. So it becomes a tool for organizing one’s thoughts and sometimes arriving at a decision on a question.
  • Export as a visual graph that can easily be shared on social media and other places.

Name: Private Idea Evaluator

Difficulty: Easy

Basic Functionality:

This app helps a user evaluate their idea with no fear of it being shared elsewhere, e.g., included in training data. The local LLM architecture can assure that all conversation about the idea stays on the user’s device. The user explains their idea, potentially in a structured way to hit certain key questions (e.g., “who would benefit from your idea?”). The LLM can provide feedback on the idea and describe similar efforts/ideas that already exist.

Example: User describes their idea as “It’s a restaurant where you can just try any breakfast cereal you want. We have all the cereals you could find commonly in a grocery store, and more esoteric cereals as well.” The LLM responds back with existing attempts at this idea like the Cereal Killer Café in London, and offers some ideas for differentiation.

Extra Feature Ideas:

  • Encrypt-at-rest storage of idea conversations for the paranoid
  • Chat sessions oriented more around brainstorming
  • Tracking variations of the idea in a graph-like structure along with pros and cons of each. E.g. “cereal buffet – all cereal is free, but you pay for milk”, “mix and match, create your own cereal”.

Category - Fun and Games

Name: Imposter

Difficulty: Medium

Basic Functionality:

In this game, you are a spy that must blend in under an assumed identity. You talk with other people who are roleplayed by the LLM. If you are sufficiently convincing, you win the game. But if you don't sound like the person you aim to portray, an LLM-controlled character will accuse you of being an imposter, and you will lose.

Extra Feature Ideas:

  • Once you have the basic gameplay working, it's easy to add more characters and scenarios.
  • Artwork with character's faces and the room is nice.

Hints:

  • You can get the LLM to speak in the voice of a character with some prompt grounding.
  • In the same grounding, explain the conditions in which the character would accuse the player of being an imposter and instruct the LLM to use the keyword "imposter" in an accusation.
  • Your chat processing code can look for the keyword "imposter" in completion output and use it to signal the "lose" condition for the game.
  • If the player successfully says X (maybe 10?) things to the LLM without triggering the "lose" condition, then that signals the "win" condition for the game.

Name: I'm Thinking Of A...

Difficulty: Easy

Basic Functionality:

In a chat session, the LLM announces "I'm thinking of a well-known fictitious character. Ask me questions and guess who it is." The user asks questions that narrow possibilities and eventually guesses.

Extra Feature Ideas:

  • Different kinds of objects (e.g., famous person, city, animal, movie) can be picked by the user to start off the game, giving it more variety.
  • Set a limit on the number of guesses the user can make or incentives to make less guesses, e.g. three stars if you can guess in less than five.
  • Let the player ask for hints if they get stumped.

Hints:

  • This can all be accomplished with grounding that tells the LLM the basic rules of the game. In the grounding, instruct the LLM to output "victory" or some keyphrase when the user has guessed the correct answer.
  • Your chat processing code can look for the victory keyword and use it to signal the "win" condition for the game.
  • The initial object that is picked could come from a pre-generated list or you could use LLM prompting to generate the object.

Name: Guess Who I Am

Difficulty: Easy

Basic Functionality:

In a chat session, the LLM announces "I am a famous person. I can't tell you my name.  Ask me questions and guess." The user asks questions that narrow possibilities and eventually guesses.

Extra Feature Ideas:

  • Have the LLM mimic the famous person by talking like them and using phrases that they would.
  • Set a limit on the number of guesses the user can make or incentives to make less guesses, e.g. three stars if you can guess in less than five.
  • Let the player ask for hints if they get stumped.

Hints:

  • This can all be accomplished with grounding that tells the LLM the basic rules of the game. In the grounding, instruct the LLM to output "victory" or some keyphrase when the user has guessed the correct answer.
  • Your chat processing code can look for the victory keyword and use it to signal the "win" condition for the game.
  • The initial person that is picked could come from a pre-generated list or you could use LLM prompting to generate it.

Name: Stump Speech

Difficulty: Medium

Basic Functionality:

The user is a corrupt politician that will pander and lie to gain popularity. They are on a campaign tour of different towns, each with its own set of people that want different things. At each stop on the tour, the user will be given a randomly generated set of issues and talking points. These will be things like "40% of population favor tax cuts" or "10% believe the resevoir's drinking water is intentionally poisoned". The user gives a short speech that panders to the audience. Their popularity increases or decreases based on how successfully they've pandered.

Extra Feature Ideas:

  • This would work well with speech recognition to let the user speak aloud like a politician. Consider using a local speech recognition capability such as Whisper or Vosk-Browser.
  • If you don't like the politics angle, the game could be rethemed in different ways. Imagine a conference room where you pitch a business idea, and people in attendance can be convinced with pandering, e.g. "Barney always likes ideas about efficiency and cost-cutting and Lanelle hates it when people use sports and team metaphors".

Hints:

  • The LLM can receive a series of prompts like "Evaluate {speech} for how it achieves {condition} and output a number between 0 (does not achieve condition at all) and 100 (fully achieves condition)". After you have a score for each condition, it's just a little more math to calculate an overall success score for the speech.
  • Avoid using the LLM for parts that don't need it so that the game runs quicker.
